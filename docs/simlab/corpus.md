## Coverage matrix (why these 78)

Every checklist item, veto, gate condition, red flag, and golden rule is
attacked by ≥2 scenarios, from both directions where a direction exists
(fire / must-not-fire). The adversarial minority is deliberate: #16
(veto right, outcome positive via takeover), #51 (weak right, +2000%
hindsight), #5 and #72 (gate lookalikes) exist so the harness learns to
score process, not outcomes, and so patches are never won cheaply.

## Harness protocol (binds R3/R4)

1. Freeze skill version; run all runnable scenarios (data-coverage
   report decides which); Opus draft → Opus verify → Haiku lineage.
2. Score process-first per the locked scoring philosophy; compute
   return-by-class aggregates separately.
3. Any FAIL → failure-ledger row → patch proposal → **owner reviews
   patch** → full-battery regression → promote or revert.
4. Bulletproof declaration requires: one full pass of ≥20 previously
   unrun scenarios, zero patches, plus return-by-class ordering
   (attractive > neutral > weak) on the aggregate with denominators
   shown.
5. Corpus grows: every future owner rejection and every live watch-loop
   surprise becomes a new registered scenario. The battery is never
   "done", it is the moat.
